## 🔹 Step-by-Step Strategy

### 1. Card Class
- Represents a single card (Suit + Rank).
- Changed from `record` → `class` for **future extensibility**:
  - Records = concise, immutable, value equality.
  - Classes = allow methods, validation, inheritance.
- **Interview Point**: “I chose class to keep flexibility for future behavior.”

---

### 2. IShuffleStrategy Interface
- Defines contract for shuffle algorithms.
- Decouples Deck from shuffle logic.
- **Interview Point**: “This follows DIP and OCP—Deck depends on abstraction, not concrete implementation.”

---

### 3. FisherYatesShuffle Class
- Implements efficient O(n) shuffle.
- Ensures uniform randomness (no bias).
- **Interview Point**: “I replaced naive shuffle with Fisher-Yates for fairness and performance.”

---

### 4. Deck Class
- Manages collection of cards.
- Methods:
  - `Reset()` → rebuilds 52 cards.
  - `Shuffle()` → delegates to injected strategy.
  - `Deal()` → removes top card, guard clause for empty deck.
  - `GetCards()` → exposes read-only list.
- **Interview Point**: “Deck encapsulates card management, uses DI for shuffle, and ensures resilience.”

---

### 5. Program.cs
- Composition root: wires dependencies.
- Injects shuffle strategy into Deck.
- Demonstrates shuffle + deal operations.
- **Interview Point**: “Program.cs separates orchestration from business logic.”

---

## 🔹 Closing Principles
- **SRP** → Each class has one responsibility.
- **OCP** → Add new shuffle strategies without modifying Deck.
- **DIP** → Deck depends on abstraction (`IShuffleStrategy`).
- **Resilience** → Guard clauses prevent runtime errors.
- **Testability** → Mock shuffle can be injected for unit tests.
